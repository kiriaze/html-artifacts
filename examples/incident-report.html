5 Whys — Root Cause Analysis
1 Why did users lose access to file history? ▶
The file_versions table was corrupted during a migration, making all version lookups fail with a database error.
2 Why was the table corrupted? ▶
Migration 0041_add_version_index dropped a write-ahead-log index before confirming the replacement was created. The WAL checkpoint failed midway, leaving the table in an inconsistent state.
3 Why did the migration run without validation? ▶
The migration script had no rollback guard. It did not check that the new index existed before proceeding to the DROP statement.
4 Why was there no rollback guard? ▶
Our migration review checklist did not require "verify new object exists before dropping old." This check existed for column changes but not index changes — a gap in the checklist introduced when index migrations were added as a pattern.
5 Why was there no bulk restore tooling? ▶
Database restores have historically been manual (1–2 per quarter). The time cost of a bulk restore script was deprioritized in planning. This incident exposed the gap: manual restore at scale took over an hour that bulk tooling would have compressed to minutes.
